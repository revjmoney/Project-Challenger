USER GUIDE

Project Challenger is a multi-model crypto paper-trading bot with a browser-based GUI. It runs up to three ML models in parallel on separate CPU cores, paper-trades each independently, and promotes the best-performing model to live trading automatically.

Data Flow

Exchange WebSocket → Feature Engine (RSI-14, ATR-14, EMA-diff, Log Return, Rolling Vol, OBI) → Ridge / XGBoost / LSTM in parallel → Paper Trade Logger → SQLite → Web GUI + SSE.

Tab 01 — Live

  • P&L cards — one per model, updated every 2 s: net P&L, trade count, win %, return %.
  • E-STOP — force-closes all open positions at the current market price and stops all workers. Requires confirmation.
  • Worker Status grid — colour-coded health for DATA_MGR, SKLEARN, XGBOOST, PYTORCH, LIVE_TRADER.
  • Recent Signals — live feed of BUY/SELL events per model, refreshed every 5 s.
  • Open Positions — in-flight paper positions with unrealised P&L.
  • Recent Trades — last 50 closed trades; filterable by model, refreshed every 15 s.

Tab 02 — Backtest

  • Set Lookback Hours (up to 8 760), Signal Threshold, and Initial Capital.
  • Coins — driven by the Backtest Coins selection in the Training tab (see below). The Backtest tab shows a live summary of what will be tested.
  • Use the PRESET buttons (SINGLE / QUICK / STANDARD / EXTENDED) to instantly load a standard coin set and lookback — presets also update the coin checkboxes.
  • Click ▶ Run Backtest — per-coin results stream in as they complete.
  • Summary shows Trades, Win%, Profit Factor, Net P&L, and Max Drawdown per model.

Tab 03 — Training

  • Coin Selection — unified checkbox panel that controls which coins are used for training, backtesting, and live/paper trading all at once. Whatever is checked here is what every part of the system uses.
    • Quick Select buttons — SINGLE · QUICK ★ · STANDARD · EXTENDED · ∞ INSANE instantly populate the checkboxes with a preset coin set.
    • DESELECT ALL — clears all ticks so you can hand-pick.
    • ⚡ RANDOM 100 — picks 100 random coins, always including BTC, ETH, LTC and DOGE.
    • CUSTOM field — type extra coins (comma-separated) to add on top of the checkbox selection.
    • SAVE COINS — persists the entire selection and restarts the live feed for all selected coins.
  • Training Tier dropdown — controls how much historical candle data is fetched per coin during training. This does not override which coins are selected.
  • The Backtest tab shows a live coin summary reflecting exactly what is checked here.
  • Hyperparameter editor covers Ridge alpha, XGBoost tree settings, and LSTM architecture + training params.
  • Save & Retrain — persists settings then triggers a full retrain from scratch.
  • Cross-validation results (direction accuracy + MSE per fold) appear after training completes.

Tab 04 — Models

  • Up to 3 archived checkpoints per engine, sorted best P&L first.
  • Status badges: active (currently running) · armed (selected for live trading) · archived.
  • Only a model that beats the current archive champion is saved after a retrain.

Tab 05 — Activity

  • Per-component status table for all 6 components with colour-coded state and last message.
  • Scrollable event log — last 500 lines from all components.

Tab 06 — Exchange & Keys

  • Select exchange: Coinbase / Binance / Kraken (restart bot to apply).
  • Coinbase API key + secret stored in data/.secrets.json with local OS protection where available.
  • Clear keys with one click to revert to PAPER mode.
  • Live Coinbase account balance auto-refreshes every 30 s when real keys are loaded.

Tab 07 — Settings

  • Paper Trading — signal threshold (min log-return to trade) and position size %.
  • Data Cache — max cache hours and refresh interval.
  • Backtest Defaults — default lookback hours pre-filled on the Backtest tab.
  • Changes apply immediately — no restart needed.

HOW MODELS WORK

All active models predict the next candle's log return. Above +SIGNAL_THRESHOLD → LONG. Below -SIGNAL_THRESHOLD → SHORT. Below threshold in either direction → HOLD.

Shared Features (all models)

log_returnln(close_t / close_{t-1})
rsi_14Relative Strength Index, 14-period
atr_14Average True Range, 14-period
ema_diff(EMA12 − EMA26) / EMA26
rolling_vol_2020-candle std dev of log returns
obiOrder Book Imbalance: (bid_vol − ask_vol) / total_vol

One StandardScaler is fitted once on the full combined training set and shared by all models and live inference.

Scientific Methodology

  • Shared CV folds — TimeSeriesSplit indices computed once, saved to data/cv_folds.pkl. Every model trains and validates on identical row ranges.
  • Sequence alignment — LSTM needs a 20-candle look-back, so the first 20 rows of each fold are unusable. All models skip those same rows so predictions occur at identical timestamps.
  • Identical normalisation — one StandardScaler fitted once on the full training set, shared by all active models.
  • Identical friction — all paper traders apply the same slippage, fee, and starting capital regardless of model.
  • Multi-coin training — features computed per coin then concatenated before fitting the scaler. Models see broader market behaviour without leaking price-scale information across coins.

Model Archive System

  1. Score — full backtest on each freshly trained model.
  2. Gate — model only archived if backtest P&L strictly beats the current best for that engine.
  3. Archive — model file saved to models/archive/<engine>/ with timestamp and P&L in the filename.
  4. Prune — only top-3 by P&L kept; older/worse files deleted.
  5. Promote — best archive entry per engine copied to the active model slot.
  6. Arm — ARMED_MODEL updated to whichever engine has the highest P&L across all archives.

TROUBLESHOOTING

Browser shows "connection refused"

The server isn't running. Start it with launcher.bat → [1] or run_web.bat.

SSE stream stops updating

Hard-reload the page. The SSE connection auto-reconnects, but a reload resets cleanly. If the server restarted, the browser reconnects within a few seconds on its own.

"No trades yet" in the Live tab

The model needs ~30 s of warm-up candles before making predictions. Lower SIGNAL_THRESHOLD in Settings if you want more signals — default is 0.008, sized to cover the fee per leg.

"No trained models" on startup

Training starts automatically when the server starts. If you cancelled it: click ⚙ Save & Retrain in the Training tab.

LSTM training is very slow (CPU-only)

Expected. Options: reduce LSTM_EPOCHS to 2–3 in the Training tab, toggle LSTM off under Active Models, or install a CUDA GPU build of PyTorch.

TUI crashes — database error

Run python -c "from database import init_db; init_db()" once to create the tables, then restart.

UnicodeEncodeError on Windows

chcp 65001
set PYTHONIOENCODING=utf-8
run_controller.bat

TUI looks garbled / wrong colours

Requires a terminal with ANSI colour support. Use Windows Terminal on Windows (not legacy cmd.exe). On Linux/macOS: any modern terminal (iTerm2, Alacritty, Kitty, etc.).

Backtest returns empty results

Candle cache may be empty. Click ↺ Data (Live tab) and wait 60–90 s for the initial cache fill, then re-run the backtest.

Models not improving across retrains

The archive gate only saves when new backtest P&L strictly beats the current champion. Check the Activity log for [ARCHIVE] ... no improvement, skipped. To force a fresh slate: delete the models/archive/ subdirectories and retrain.

Insane tier training takes forever

The insane tier fetches candles for every coin on the exchange (potentially 100+). Use extended (10 coins) for a good coverage/speed tradeoff. Click ↻ Refresh List before switching to insane.

Coinbase WebSocket keeps disconnecting

Normal — automatic reconnect with 5-second backoff is built in. The PAPER stream never disconnects.

API key errors after saving new keys

After saving keys in the Exchange & Keys tab, restart the bot (■ Stop → ▶ Start) so the WebSocket connection picks up the new credentials.

psutil cpu_affinity warning on startup

Non-fatal. Some VMs and containers don't expose the CPU affinity API. The bot runs normally without core pinning.

TRADING TERMS

Plain-English definitions for terms shown in the GUI and used throughout the system.

Signals & Decisions

SignalA model's output decision for a given candle — either BUY, SELL, or HOLD. Generated whenever a new 15-minute candle closes.
BUY signalThe model predicts price will rise. A paper long position is opened at the current market price.
SELL signalThe model predicts price will fall. An open long position is closed (or a short is entered, depending on config).
HOLDThe model's predicted return is below the signal threshold — not confident enough to trade.
Signal ThresholdMinimum predicted log-return required to trigger a BUY or SELL. Default: 0.0002. Higher values = fewer, higher-confidence trades.
Log ReturnThe natural log of the price ratio between two candles: ln(P₂/P₁). Used as the model's prediction target because it's symmetric and stationary.

P&L & Performance

P&LProfit and Loss — the net dollar gain or loss across all closed trades for a model since the bot was started.
Unrealised P&LThe paper gain or loss on a position that is still open, calculated at the current market price.
Win %Percentage of closed trades that were profitable. A 50% win rate with good risk management can still be profitable if winners are larger than losers.
Return %Net P&L as a percentage of the starting capital allocated to that model.
Profit FactorGross profit divided by gross loss. A value above 1.0 means the model is profitable overall. Values above 1.5 are considered good.
Max DrawdownThe largest peak-to-trough decline in cumulative P&L during the backtest period, expressed as a percentage. Lower is better.
Net P&LTotal profit minus total losses minus all trading fees.

Trading Mechanics

Paper TradingSimulated trading using real market prices but no actual money. Positions are tracked in the local database. Used to evaluate model performance safely.
Open PositionA trade that has been entered but not yet closed. The bot is holding a simulated quantity of the coin.
Position Size %What fraction of the model's allocated capital is used per trade. Set in Settings → Paper Trading.
FeeSimulated exchange trading fee applied to each trade. Defaults to Coinbase's taker fee rate.
E-STOPEmergency Stop. Force-closes all open positions at the current market price immediately and stops all workers. All closing trades are written to the database.
CandleA 15-minute OHLCV (Open, High, Low, Close, Volume) data point. All models predict on the same candle timestamps for a fair comparison.

Models

Ridge (RIDGE)Linear regression with L2 regularisation. The fastest model — good baseline. Predicts price direction from scaled features.
XGBoost (XGBOOST)Gradient-boosted decision trees. Captures non-linear feature relationships. Slower than Ridge but typically more accurate.
LSTM (LSTM)Long Short-Term Memory recurrent neural network (PyTorch). Processes sequences of 32 candles to capture temporal patterns. Slowest but best at trend recognition. Auto-uses CUDA if available.
LightGBM (LGBM)Fast gradient-boosted trees from Microsoft. Available via the generic worker for experimentation.
CatBoost (CATBOOST)Gradient boosting from Yandex, handles categorical features natively. Available via the generic worker.
ARIMAAutoRegressive Integrated Moving Average. Classical statistical time-series model. Refits every 20 ticks on log returns.
ProphetMeta's time-series forecasting library. Refits every 50 ticks. Forecasts one candle (15 min) ahead.

Training & Validation

Cross-Validation (CV)A method of evaluating model quality by training and testing on multiple non-overlapping time windows. More reliable than a single train/test split.
CV FoldOne train/test split within cross-validation. The number of folds is set in Settings → Training Defaults.
Direction AccuracyHow often the model correctly predicted the direction (up or down) of the next candle across CV folds. Shown on the Models and Training tabs.
MSEMean Squared Error — average squared difference between predicted and actual log returns. Lower is better.
Training TierHow many coins are included in the training dataset. More coins = more generalised model but longer training time.
Model ArchiveThe top 3 saved checkpoints per model engine, ranked by backtest P&L. Only a model that beats the current best is saved after a retrain.
Armed modelThe checkpoint currently selected for live/paper trading. Automatically set to whichever archived model has the highest backtest P&L.

Features

RSI-14Relative Strength Index over 14 candles. Measures momentum — values above 70 indicate overbought, below 30 indicate oversold.
ATR-14Average True Range over 14 candles. Measures market volatility — how much the price typically moves per candle.
EMA-diffDifference between fast and slow Exponential Moving Averages. Positive values indicate upward momentum.
Rolling VolRolling standard deviation of log returns. Higher values = higher recent price volatility.
OBIOrder Book Imbalance. The ratio of bid vs ask volume from the exchange's order book. Positive = more buy pressure.

System Terms

PAPER modeBot runs simulated trades without placing real orders. No account API keys required.
WorkerA separate OS process running one model. Each worker has its own memory space and paper trader, receiving price ticks via a shared queue.
SSEServer-Sent Events — the mechanism the GUI uses to receive live updates from the backend without polling. One persistent HTTP connection streams data as it arrives.
DATA_MGRThe data manager component — connects to the exchange WebSocket, computes features from incoming candles, and broadcasts ticks to all model workers.
LIVE_TRADERComponent that monitors model signals and, when live trading is armed, would submit real orders to the exchange. In paper mode it logs virtual trades only.
LookbackHow many hours (training) or days (backtest config) of historical data to fetch from the exchange for training or replay.
